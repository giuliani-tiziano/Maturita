Progettazione DB
concettuale, logica e fisica

Dal problema reale al database MySQL: tre fasi, un metodo.

Le tre fasi

Il processo di progettazione

Fase 1 — Concettuale
Obiettivo: capire il problema. Strumento: diagramma E/R. Output: schema E/R con entità, attributi, associazioni, cardinalità. Indipendente da qualsiasi DBMS.
Fase 2 — Logica
Obiettivo: tradurre in tabelle. Strumento: regole di traduzione E/R → relazionale. Output: schema relazionale con PK e FK. Normalizzazione fino alla 3NF.
Fase 3 — Fisica
Obiettivo: implementare in MySQL. Strumento: SQL + phpMyAdmin/HeidiSQL. Output: CREATE TABLE con tipi di dato, vincoli e indici ottimizzati.
«Una buona progettazione concettuale vale ore di debugging. Un diagramma E/R sbagliato produce un database sbagliato.» — Principio di progettazione
Sezione 02

Diagramma Entità/Relazione (E/R)

ElementoSimboloDescrizione
EntitàRettangoloOggetto del mondo reale con esistenza autonoma. Es. Studente, Corso, Prodotto.
AttributoEllisseProprietà di un'entità. Es. nome, cognome, data_nascita.
ChiaveEllisse sottolineataAttributo che identifica univocamente le istanze dell'entità.
AssociazioneRomboRelazione tra due o più entità. Es. FREQUENTA tra Studente e Corso.
CardinalitàCoppie (min,max)Quante istanze partecipano all'associazione: (0,1), (1,1), (0,N), (1,N).

La cardinalità esprime quante istanze di ciascuna entità possono partecipare all'associazione:

TipoSignificatoEsempio
1:1Un'istanza di A è associata ad al massimo una di B e viceversaDipendente ↔ Scrivania (ognuno ha una)
1:NUn'istanza di A può essere associata a molte di B; ogni B a una sola ADipartimento → N Dipendenti
N:NUn'istanza di A a molte B; un'istanza di B a molte AStudenti ↔ Corsi

La partecipazione può essere obbligatoria (ogni istanza DEVE partecipare) o opzionale (può non partecipare, min=0).

  • Binaria: coinvolge due entità distinte. Tipo più comune. Es. Studente FREQUENTA Corso.
  • Unaria (ricorsiva): un'entità associata con se stessa con ruoli diversi. Es. Dipendente GESTISCE Dipendente (un dipendente può essere manager di altri).
  • Ternaria (o multipla): coinvolge tre o più entità. Es. Fornitore FORNISCE Prodotto a Negozio.

Un'entità debole non ha attributi sufficienti per identificarsi da sola. Dipende da un'entità forte tramite una relazione identificante.

Si rappresenta con un doppio rettangolo. La sua chiave primaria è composta: include la PK dell'entità forte + un attributo discriminante.

Esempio: Riga_Ordine è entità debole — una riga d'ordine si identifica solo con il numero d'ordine (entità forte Ordine) + numero di riga. Senza l'ordine a cui appartiene, la riga non ha senso.

ISA (Is-A): una generalizzazione dove le entità figlie "sono un tipo di" entità padre. Le figlie ereditano tutti gli attributi del padre e aggiungono i propri.

Tipi di gerarchia:

  • Totale: ogni istanza padre è necessariamente in almeno una figlia
  • Parziale (subset): alcune istanze padre non appartengono a nessuna figlia
  • Esclusiva: ogni istanza è in al massimo una figlia
  • Sovrapposta: un'istanza può appartenere a più figlie contemporaneamente

Ristrutturazione E/R: prima della traduzione logica si elimina: attributi multivalore (→ nuova entità), generalizzazioni ISA (vedi strategie di traduzione), associazioni complesse.

Sezione 03

Progettazione Logica — da E/R a Relazionale

Lo schema logico traduce il diagramma E/R in tabelle (relazioni) con colonne, chiavi primarie e chiavi esterne.

Elemento E/RTraduzione nel modello relazionale
Entità forteUna tabella con PK (la chiave dell'entità)
Associazione 1:1FK nella tabella con partecipazione obbligatoria (o fusione se entrambe obbligatorie)
Associazione 1:NFK nell'entità "molti" che referenzia la PK dell'entità "uno" — nessuna tabella aggiuntiva
Associazione N:NTabella associativa (ponte) con PK composta = FK delle due entità; attributi dell'associazione vanno in questa tabella
Entità deboleTabella con PK = discriminante + FK all'entità forte
Generalizzazione ISA — unica tabellaUn'unica tabella con tutti gli attributi + colonna tipo; semplice ma con molti NULL
ISA — padre + figlieTabella padre + tabelle figlie con PK = FK al padre; più flessibile
ISA — solo figlieSolo tabelle figlie con attributi padre replicati; no JOIN ma ridondanza
Attributo multivaloreTabella separata con FK all'entità e il valore come colonna

Una dipendenza funzionale A → B significa: dato un valore di A, esiste uno e un solo valore di B corrispondente. A è il determinante, B il dipendente.

Le anomalie emergono quando uno schema non è ben normalizzato:

  • Anomalia di inserimento: impossibile inserire dati senza informazioni ridondanti obbligatorie. Es. non si può inserire un nuovo prodotto senza avere almeno un ordine.
  • Anomalia di cancellazione: cancellando un record si perdono accidentalmente altre informazioni. Es. eliminando l'unico ordine di un fornitore si perdono i suoi dati.
  • Anomalia di aggiornamento: un dato è ripetuto in più righe; modificandone solo alcune si crea inconsistenza. Es. cambio indirizzo cliente in una riga su dieci.

La soluzione alle anomalie è la normalizzazione.

Sezione 04

Normalizzazione — 1NF, 2NF, 3NF

Processo di ristrutturazione per eliminare ridondanze e anomalie.

Regola: ogni attributo deve essere atomico (un solo valore, non divisibile). Nessun gruppo ripetuto. Deve esistere una chiave primaria.

Viola la 1NF:

IDNomeTelefoni
1Mario333-111, 02-9999

Telefoni contiene più valori → non atomico.

Rispetta la 1NF:

IDNomeTelefono
1Mario333-111
1Mario02-9999

Regola: soddisfa 1NF + ogni attributo non-chiave dipende dall'intera chiave primaria (nessuna dipendenza parziale). Si applica quando la PK è composta.

Dipendenza parziale: un attributo non-chiave dipende da parte della PK composta.

Viola la 2NF:

id_ordid_prodqtanome_prod
1103Penna

PK = {id_ord, id_prod}. Ma nome_prod dipende solo da id_prod → dipendenza parziale.

Rispetta la 2NF:

id_ordid_prodqta
1103
id_prodnome_prod
10Penna

Regola: soddisfa 2NF + nessun attributo non-chiave dipende transitivamente da un altro attributo non-chiave (nessuna dipendenza transitiva).

Dipendenza transitiva: A → B e B → C implica A → C, ma B non è chiave.

Viola la 3NF:

id_dipid_repnome_rep
15IT

id_dip → id_rep → nome_rep. nome_rep dipende da id_rep, non direttamente dalla PK.

Rispetta la 3NF:

id_dipid_rep
15
id_repnome_rep
5IT
Sezione 05

Progettazione Fisica — Implementazione MySQL

La progettazione fisica traduce lo schema logico in SQL concreto, scegliendo tipi di dato, vincoli e strategie di ottimizzazione.

Cosa rappresentaTipo consigliatoMotivo
Identificatori auto-incrementaliINT UNSIGNED AUTO_INCREMENTSenza segno → range doppio (0 a 4 miliardi)
ID di tabelle grandiBIGINT UNSIGNEDPer sistemi con miliardi di record
Testo corto (nome, città)VARCHAR(n)n = lunghezza massima prevista + margine
Testo lungo (descrizione, bio)TEXTNo indice diretto; per testi > 255 char
Importi monetariDECIMAL(10,2)MAI FLOAT (arrotondamenti!); precisione esatta
Solo dataDATEYYYY-MM-DD; più efficiente di DATETIME
Data e oraDATETIMEo TIMESTAMP (auto-aggiornamento, fuso orario)
Flag sì/noTINYINT(1) o BOOLEAN0=false, 1=true
Valori da lista fissaENUM('A','B','C')Efficiente, ma difficile da modificare

Schema logico: Studente(id, nome, cognome, email) — Corso(id, titolo, ore) — Iscrizione(id_stud FK, id_corso FK, data_iscr, voto_finale)

CREATE TABLE Studenti (
    id        INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    nome      VARCHAR(50)  NOT NULL,
    cognome   VARCHAR(50)  NOT NULL,
    email     VARCHAR(100) NOT NULL UNIQUE,
    data_nasc DATE,
    creato_il TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE Corsi (
    id     INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    titolo VARCHAR(100) NOT NULL,
    ore    SMALLINT     NOT NULL CHECK (ore > 0),
    attivo TINYINT(1)   NOT NULL DEFAULT 1
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE Iscrizioni (
    id_studente INT UNSIGNED NOT NULL,
    id_corso    INT UNSIGNED NOT NULL,
    data_iscr   DATE         NOT NULL DEFAULT (CURDATE()),
    voto_finale DECIMAL(5,2) CHECK (voto_finale BETWEEN 18 AND 30),
    PRIMARY KEY (id_studente, id_corso),
    FOREIGN KEY (id_studente) REFERENCES Studenti(id) ON DELETE CASCADE,
    FOREIGN KEY (id_corso)    REFERENCES Corsi(id)    ON DELETE RESTRICT
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Indici per ottimizzare le query frequenti
CREATE INDEX idx_iscrizioni_corso ON Iscrizioni(id_corso);
CREATE INDEX idx_studenti_cognome ON Studenti(cognome);
