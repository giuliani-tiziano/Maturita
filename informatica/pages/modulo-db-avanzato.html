Oltre le basi SQL

Un database non serve solo a memorizzare e leggere dati: deve garantire consistenza anche in caso di errori, gestire accessi concorrenti, essere veloce con milioni di righe, e sopravvivere a guasti. Questo modulo copre gli strumenti avanzati per questi obiettivi.

Le quattro proprietà ACID

A
Atomicità
La transazione è indivisibile: tutto o niente. Un errore a metà → ROLLBACK automatico.
C
Consistenza
La transazione porta il DB da uno stato valido a un altro, rispettando tutti i vincoli.
I
Isolamento
Le transazioni concorrenti non si "vedono" a vicenda durante l'esecuzione.
D
Durabilità
Dopo COMMIT i dati persistono anche dopo crash (grazie al transaction log).
Modulo 10 · ACID

Transazioni
ACID

BEGIN, COMMIT, ROLLBACK
-- Esempio: trasferimento bancario
-- SENZA transazione: se il server crasha dopo
-- il prelievo ma prima del deposito → denaro perso!

START TRANSACTION;  -- o BEGIN

  -- Preleva da conto A
  UPDATE conti
  SET saldo = saldo - 100
  WHERE id = 1;

  -- Deposita su conto B
  UPDATE conti
  SET saldo = saldo + 100
  WHERE id = 2;

COMMIT;  -- conferma entrambe le modifiche

-- In caso di errore:
-- ROLLBACK;  -- annulla tutto, nessuna modifica
// PHP con transazioni MySQLi
$conn->begin_transaction(); // o query("BEGIN")

try {
    $stmt1 = $conn->prepare(
        "UPDATE conti SET saldo=saldo-? WHERE id=?"
    );
    $stmt1->bind_param('ii', $importo, $id_da);
    $stmt1->execute();

    $stmt2 = $conn->prepare(
        "UPDATE conti SET saldo=saldo+? WHERE id=?"
    );
    $stmt2->bind_param('ii', $importo, $id_a);
    $stmt2->execute();

    $conn->commit(); // tutto ok → salva
    echo json_encode(['successo' => true]);

} catch (Exception $e) {
    $conn->rollback(); // errore → annulla
    http_response_code(500);
    echo json_encode(['errore' => $e->getMessage()]);
}
Deadlock

Due transazioni si bloccano a vicenda aspettando risorse che l'altra tiene bloccate:

-- T1 blocca Riga A, attende Riga B
-- T2 blocca Riga B, attende Riga A
-- → DEADLOCK! MySQL fa ROLLBACK del "victim"

MySQL rilevazione automatica: il DBMS riconosce il ciclo e fa ROLLBACK della transazione "più giovane".

Prevenzione dei deadlock

  • Accedere sempre alle risorse nello stesso ordine
  • Tenere le transazioni il più brevi possibile
  • Usare SELECT ... FOR UPDATE solo dove necessario
  • Ridurre il livello di isolamento se possibile
  • Implementare retry logic nel codice applicativo
Modulo 10 · Performance

Indici
e EXPLAIN

Indici B-tree

Un indice è come l'indice analitico di un libro: invece di scorrere tutte le pagine (full table scan), MySQL salta direttamente alla posizione giusta.

-- Creare un indice
CREATE INDEX idx_cognome
ON Studenti(cognome);

-- Indice su più colonne (compound index)
CREATE INDEX idx_nome_cognome
ON Studenti(cognome, nome);

-- Rimuovere un indice
DROP INDEX idx_cognome ON Studenti;

-- Gli indici su PK e FK sono creati automaticamente
-- da InnoDB (non vanno creati manualmente)

Quando creare un indice

SÌ: colonne usate frequentemente in WHERE, ORDER BY, GROUP BY, JOIN ON; alta cardinalità (es. email, codice fiscale).

NO: bassa selettività (es. genere M/F — solo 2 valori); tabelle piccole; colonne raramente cercate; tabelle con molti INSERT/UPDATE frequenti.

Impatto performance

Letture più veloci, scritture più lente. Ogni INSERT/UPDATE/DELETE deve aggiornare anche le strutture indice. Troppi indici = overhead inutile.

EXPLAIN — leggere il query plan
-- Aggiungere EXPLAIN davanti a qualsiasi SELECT:
EXPLAIN SELECT s.nome, s.cognome, c.titolo
FROM Studenti s
JOIN Iscrizioni i ON s.id = i.id_studente
JOIN Corsi c ON i.id_corso = c.id
WHERE s.cognome = 'Rossi';

-- Output (colonne più importanti):
-- type:    ALL (pericoloso!), ref, eq_ref, const, index
-- key:     quale indice usa (NULL se nessuno!)
-- rows:    stima righe esaminate (più basso = meglio)
-- Extra:   Using filesort (lento!), Using temporary (lento!),
--          Using index (veloce — index-only scan)

Interpretare EXPLAIN

  • type=ALL + rows alta: full table scan su tabella grande → aggiungere un indice
  • key=NULL: MySQL non usa nessun indice → creare indice sulla colonna nel WHERE
  • Using filesort: ordinamento non supportato dall'indice → creare indice su colonna ORDER BY
  • type=const o eq_ref: lookup su PK o UNIQUE → ottimale, niente da fare
Modulo 10 · Affidabilità

Backup
e ripristino

mysqldump
# Backup completo del database
mysqldump -u root -p nomeDB > backup.sql

# Backup di tabelle specifiche
mysqldump -u root -p nomeDB tabella1 tabella2 > partial.sql

# Backup compresso (consigliato)
mysqldump -u root -p nomeDB | gzip > backup.sql.gz

# Backup di tutti i database
mysqldump -u root -p --all-databases > all_dbs.sql

# Ripristino
mysql -u root -p nomeDB < backup.sql

# Ripristino da compresso
gunzip < backup.sql.gz | mysql -u root -p nomeDB

Contenuto del file .sql

mysqldump produce un file SQL testuale leggibile con: istruzioni DROP TABLE IF EXISTS, CREATE TABLE con la struttura, INSERT INTO per tutti i dati. È sia schema che dati in un unico file.

Strategie di backup

  • Full backup: copia tutto il DB → mysqldump
  • Incremental: solo le modifiche dall'ultimo backup → binary log
  • 3-2-1 rule: 3 copie, 2 supporti diversi, 1 offsite
  • Pianificare backup automatici con cron
  • Testare regolarmente il ripristino!

MongoDB — struttura e query base
// MongoDB usa BSON (Binary JSON)
// Database → Collections → Documents

// Document di esempio nella collection "utenti":
{
  "_id": ObjectId("507f1f77bcf86cd799439011"),
  "nome": "Mario Rossi",
  "email": "mario@example.com",
  "eta": 25,
  "hobby": ["calcio", "lettura"],  // array
  "indirizzo": {                    // documento annidato
    "via": "Via Roma 1",
    "citta": "Milano"
  }
}

// I campi possono variare tra documenti
// nella stessa collection!
// Query MongoDB (sintassi JavaScript)
db.utenti.find({ eta: { $gt: 20 } })
// equivalente SQL: SELECT * WHERE eta > 20

db.utenti.findOne({ email: "mario@..." })
// equivalente: SELECT * WHERE email=... LIMIT 1

db.utenti.insertOne({
  nome: "Luigi",
  email: "luigi@example.com"
})

db.utenti.updateOne(
  { _id: ObjectId("507f...") },
  { $set: { eta: 26 } }
)

db.utenti.deleteOne({ _id: ObjectId("507f...") })
Modulo 10 · Architetture

ORM
e Scalabilità

ORM — Object-Relational Mapper

Un ORM mappa gli oggetti del linguaggio di programmazione alle tabelle del database, astraendo SQL.

// Senza ORM (SQL diretto):
$stmt = $conn->prepare("SELECT * FROM utenti WHERE id=?");
$stmt->bind_param('i', $id);
$stmt->execute();
$user = $stmt->get_result()->fetch_assoc();

// Con ORM (es. Eloquent in Laravel):
$user = User::find($id);
$user->nome = 'Mario';
$user->save(); // genera automaticamente UPDATE SQL

// Relazioni:
$ordini = $user->ordini; // lazy loading automatico

Pro e contro degli ORM

Pro: sviluppo più rapido, meno boilerplate, portabilità tra DB, gestione automatica delle relazioni.

Contro: query auto-generate possono essere inefficienti (N+1 problem), astrazione può nascondere problemi di performance, curva di apprendimento.

Scalabilità

Verticale (Scale Up)

Aggiungere più CPU, RAM o storage al server esistente. Semplice ma ha un limite fisico e costi crescenti. Tradizionalmente usata con DB relazionali.

Orizzontale (Scale Out)

Aggiungere più server. Richiede distribuzione dei dati. MongoDB e sistemi NoSQL sono progettati per questo. I DB relazionali possono scalare orizzontalmente con sharding o cluster.

Replicazione Master-Slave

Il master gestisce le scritture. Gli slave ricevono copie sincronizzate e gestiscono le letture. Vantaggi: alta disponibilità, bilanciamento del carico in lettura, failover automatico.
