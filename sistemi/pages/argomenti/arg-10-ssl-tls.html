SSL/TLS
HTTPS e Handshake

Il protocollo che rende sicure le comunicazioni su Internet.

SSL/TLS è un protocollo crittografico che si inserisce tra il livello TCP e il livello applicativo (HTTP, FTP, SMTP…). Garantisce confidenzialità (cifratura), autenticazione (certificati) e integrità (MAC).

SSL vs TLS

  • SSL 2.0, SSL 3.0: deprecati per vulnerabilità gravi.
  • TLS 1.0, TLS 1.1: deprecati (dal 2021 disabilitati nei browser).
  • TLS 1.2: ancora ampiamente usato.
  • TLS 1.3: standard attuale (2018) — più sicuro e più veloce.

Cipher Suite

La cipher suite definisce l'insieme di algoritmi usati in SSL/TLS: scambio chiavi (RSA, DHE, ECDHE) + cifratura simmetrica (AES-256-GCM, ChaCha20) + hash per MAC (SHA-256, SHA-384). Negoziata durante l'handshake.

01

SSL/TLS Handshake

Passi dettagliati dell'Handshake

  1. CLIENT HELLO: il client invia le cipher suite supportate + nonce client (byte casuali).
  2. SERVER HELLO: il server sceglie la cipher suite + invia nonce server.
  3. CERTIFICATO: il server invia il proprio certificato digitale X.509.
  4. (opzionale) RICHIESTA CERTIFICATO CLIENT: il server può richiedere autenticazione mutual.
  5. SERVER HELLO DONE: il server segnala la fine delle proposte.
  6. VERIFICA CERTIFICATO: il client verifica validità, CA fidata, firma, corrispondenza dominio.
  7. PRE-MASTER SECRET: il client genera bytes casuali, li cifra con la chiave pubblica del server (dal certificato), li invia.
  8. DECIFRATURA: il server decifra il pre-master secret con la propria chiave privata.
  9. MASTER SECRET: entrambi calcolano indipendentemente: master_secret = f(pre_master_secret + nonce_client + nonce_server)
  10. 4 CHIAVI DERIVATE dal master secret (vedi sezione successiva).
  11. CHANGE CIPHER SPEC + FINISHED: entrambi comunicano il passaggio alla comunicazione cifrata e si inviano il primo messaggio cifrato per verifica.

Sistema Ibrido in SSL/TLS

L'handshake usa la crittografia asimmetrica (RSA o DH) solo per scambiare il pre-master secret. Tutta la comunicazione successiva usa AES con le chiavi di sessione. Combina sicurezza e prestazioni.

02

Le 4 Chiavi SSL/TLS

Dal master secret vengono derivate 4 chiavi di sessione. La separazione per direzione aumenta la sicurezza.

  • Chiave di cifratura Client → Server: cifra i dati inviati dal client al server.
  • Chiave di cifratura Server → Client: cifra i dati inviati dal server al client.
  • Chiave MAC Client → Server: garantisce integrità dei dati inviati dal client.
  • Chiave MAC Server → Client: garantisce integrità dei dati inviati dal server.

Perché 4 e non 1? Se un attaccante compromette una chiave di cifratura, non ottiene automaticamente la chiave MAC. La separazione tra funzioni e direzioni riduce il danno da una singola compromissione.

Nonce — Numero usato una sola volta

Il nonce client e il nonce server sono inclusi nel calcolo del master secret. Questo garantisce che ogni sessione SSL/TLS abbia chiavi di sessione uniche, anche tra gli stessi due endpoint. Previene gli attacchi replay.

03

SSL Record Layer

Il Record Layer è il livello inferiore di SSL/TLS che gestisce il trasporto effettivo dei dati.

Funzionamento

  1. Riceve i dati dall'applicazione (es. HTTP).
  2. Frammenta in blocchi di dimensione massima.
  3. Comprime (opzionale, raramente usato).
  4. Aggiunge il MAC (calcolato con la chiave MAC di sessione) per l'integrità.
  5. Cifra con la chiave di cifratura di sessione.
  6. Aggiunge l'header SSL Record.
  7. Invia via TCP.

Il destinatario esegue il processo inverso: decifra, verifica il MAC, riassembla.
